## 目的

インベントリ＆装備システムの /simplify レビューで指摘された改善点のうち、
現時点では修正コストに見合わないためスキップした項目をまとめる。
将来的にコードベースが成長して重複が顕著になった場合に対応を検討する。

## 優先度

C（低 — 機能に影響なし、改善の余地があるリファクタ候補）

## スキップした指摘一覧

### 1. JSON セーブ/ロードのボイラープレート共通化

- **対象:** `achievement_tracker.gd`, `reward_manager.gd`, `inventory_manager.gd`
- **問題:** 3ファイルで `_save()` / `_load()` が同じ JSON 読み書きパターンを繰り返している
- **見送り理由:** 3ファイル横断の大改修。共通ヘルパーを導入するとファイル間の依存が増える

### 2. Database + def_map パターン共通化

- **対象:** `achievement_tracker.gd`, `reward_manager.gd`, `inventory_manager.gd`
- **問題:** `_database` + `_def_map` の構築パターンが3ファイルで重複
- **見送り理由:** 同上。基底クラスまたはユーティリティの追加が必要

### 3. `_gui_input` 左クリックハンドラ共通化

- **対象:** `inventory_list_item.gd`, `equip_slot_panel.gd`, `achievement_list_item.gd`, `reward_list_item.gd`
- **問題:** 4ファイルで左クリック検出 → シグナル発火の同一パターン
- **見送り理由:** 基底クラス追加が必要。各ファイルのロジックは数行で、抽象化のメリットが薄い

### 4. EquipmentStatCache と RewardEffectCache の統合

- **対象:** `equipment_stat_cache.gd`, `reward_effect_cache.gd`
- **問題:** 両方とも「数値フィールドの集約キャッシュ」という同じ役割
- **見送り理由:** アーキテクチャ変更。フィールド名・用途が異なるため統合すると可読性が下がる

### 5. タブ間のリスト構築/選択パターン共通化

- **対象:** `equipment_tab.gd`, `achievement_tab.gd`, `reward_tab.gd`
- **問題:** リスト再構築 → 項目追加 → 選択状態復元の流れが3タブで類似
- **見送り理由:** 3タブ横断の改修。各タブの表示内容・フィルタ条件が異なり、共通化が複雑

### 6. `get_effective_*` 6メソッドのヘルパー統合

- **対象:** `pawn.gd` の `get_effective_max_hp()` 等6メソッド
- **問題:** 「基礎値 + 装備補正 → 報酬%乗算」のパターンが繰り返されている
- **見送り理由:** 各メソッドの計算式が微妙に異なる（固定値加算 vs %加算）。ヘルパー化すると可読性が下がる

### 7. `_current_items` 配列の `_list_container.get_children()` 置換

- **対象:** `equipment_tab.gd`
- **問題:** `_current_items` を別途管理せず、子ノードから直接取得できる
- **見送り理由:** 子ノードに空メッセージ Label も含まれるため、型フィルタが必要になり冗長

### 8. `item_used` シグナルの `id` パラメータ削除

- **対象:** `inventory_manager.gd` の `item_used` シグナル
- **問題:** `definition` から `id` を取得できるため `id` パラメータが冗長
- **見送り理由:** シグナル互換性。既存の接続先を全て変更する必要がある

### 9. `get_equip_cache()` の不変参照化

- **対象:** `inventory_manager.gd`
- **問題:** 外部から `_equip_cache` の値を書き換え可能
- **見送り理由:** GDScript に read-only wrapper がないため、実現には別のアプローチが必要

### 10. `EquipmentStatCache` 廃止してオンデマンド計算

- **対象:** `equipment_stat_cache.gd`, `inventory_manager.gd`, `pawn.gd`
- **問題:** キャッシュを事前計算する代わりに、毎回装備から直接計算する方がシンプル
- **見送り理由:** RewardEffectCache と同じパターンを踏襲しており、統一性を維持するため

## 対応方針

- 個別に着手するのではなく、セーブ/ロード統合タスク（`セーブロード統合.md`）と合わせて検討する
- 項目 1, 2 はセーブ統合時に自然に解消される可能性が高い
- 残りの項目は、同じファイルに別の変更を加える際に「ついでに」対応するのが効率的
